---
title: "Quickstart: Debug Your First .NET Assembly with dnSpy"
description: "Load a .NET assembly into dnSpy, navigate its types and methods, set a breakpoint, and start a debugging session — no source code needed."
---

This guide walks you through the most common dnSpy workflow: opening a compiled .NET assembly, browsing its code, and attaching the debugger to inspect it at runtime. You will be up and running in about five minutes.

<Steps>
  <Step title="Open an assembly">
    Launch `dnSpy.exe`. To load an assembly, do one of the following:

    - Go to **File > Open** and select a `.dll` or `.exe` file.
    - Drag and drop a `.dll` or `.exe` file directly onto the dnSpy window.

    The assembly appears in the **Assembly Explorer** panel on the left side of the screen.
  </Step>
  <Step title="Navigate the Assembly Explorer">
    Expand the assembly node in **Assembly Explorer** to browse its structure. The tree shows:

    - Namespaces
    - Types (classes, interfaces, enums, structs)
    - Members (methods, properties, fields, events)

    Double-click any method or type to open it in the decompiler view. dnSpy decompiles the IL back to C# (or Visual Basic) automatically.
  </Step>
  <Step title="Set a breakpoint">
    With a method open in the decompiler view, click in the left gutter (the grey bar to the left of the line numbers) next to any statement. A red circle appears to indicate an active breakpoint.

    You can also right-click a line and choose **Add Breakpoint** from the context menu.

    <Tip>
      Use **Ctrl+Shift+F** to search all loaded assemblies for a type name, method name, or string literal. This is the fastest way to find the code you want to break on.
    </Tip>
  </Step>
  <Step title="Start debugging">
    Go to **Debug > Start Debugging** (or press **F5**). dnSpy will prompt you to select the executable to launch, or you can go to **Debug > Attach to Process** to attach to an already-running process.

    Once the process starts, dnSpy breaks execution when it hits your breakpoint and highlights the current statement in the decompiler view.
  </Step>
  <Step title="Inspect variables">
    While paused at a breakpoint, use the bottom panels to inspect program state:

    - **Locals** — shows all local variables in the current stack frame
    - **Watch** — lets you add expressions to evaluate continuously
    - **Autos** — shows variables related to the current and previous statement
    - **Call Stack** — shows the full call chain; double-click a frame to navigate to it

    You can type C# or Visual Basic expressions directly into the **Watch** window to evaluate them against the live process.

    When you are done inspecting, press **F5** to continue execution, **F10** to step over the next statement, or **F11** to step into a called method.
  </Step>
</Steps>
